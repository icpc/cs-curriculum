Read Chapter 5: Complete Search from the text. This discusses various ways of covering a search space and will lead us into a discussion of different algorithm classes
This week we are working in teams of 3. Do the initial analysis together and then start to learn how to schedule the keyboard / screen. Figure out how many of the problems each can solve individually and overlap design with coding to make better use of the limited resource (the keyboard / screen). Once you have exhausted the problems that can be done individually, work together to design, code, test, and debug the harder problems.

Working on teams of three requires more attention paid to budgeting the keyboard / screen resource effectively. Practice switching from problem to problem as coding is finished and tests need to be written or bugs need to be found. Don't leave two people idle while one person stares at the screen. Learn to read and debug someone else's code.

While you do these problems, keep good records on how much time you are spending in the following areas:

Analysis

  • Understanding (What do they want?)
  • Recognition (Ahh! That's what they want and here's how I'll do it)
  • Ranking (trivial, easy, medium, hard, impossible)
  • Sizing (time / space complexity)
  • Time Estimate (how long to code up)
  • Ordering (how would you order these problems for solving)

The Problems

  1. Analysis (see above)
  2. Coding (just the physical typing of code)
  3. Testing (setting up and running tests)
  4. Debugging (time spent fixing problems found in Testing)
  5. Re-coding (any time spent re-designing / re-writing you solution. Does not include time fixing bugs)
  6. Total time to solution (including steps 2-5 above)
  7. Record the number of failed submissions
